# Тестовое задание Yii2 + Postgres

### Апдейт
Все изменения находятся в одном коммите для удобства проверки.

Проведена работа по следующими проблемам:

```
Работа с Docker. Конфигурация требует доработки: из-за конфликтов портов и отсутствия очередности запуска сервисов (depends_on) проект не стартует автоматически. Для нас этот навык является важным, так как мы ожидаем от специалиста умения создавать автономную среду, которая запускается «в одну команду» без ручного исправления.
```
Добавил `depends_on` с условием `service_healthy`, теперь апишка стартанет только после того как пройдет первый healthchek бдшки. Внешние порты можно настроить в `.env` следующими переменными: `DB_EXTERNAL_PORT` и `YII_EXTERNAL_PORT`. Подозреваю, что при прошлой проверке дефолтные порты на хосте были заняты, поэтому и не запустилось.

```
Опасный момент в UPDATE. В запросе на обновление данных пропущено условие WHERE. Сейчас этот код не активен, но если его запустить в будущем, он может случайно стереть данные во всей таблице.
```
Добавил аргумент `id` в  `WriteMapperTrait::update()`, теперь изменить данные можно только для конкретной сущности (включая изменение `id`). У данной имплементации есть ограничение - нельзя изменять значения массово.

```
Баг в логике. Счетчик общего количества элементов (totalItems) всегда выдает единицу, независимо от реальности.
```
Добавил from() в запрос с подсчетом количества строк.

```
Валидация данных. Недостаточная фильтрация входных параметров. Пропускаются некорректные значения (отрицательная цена/пробег, будущие даты), а также отсутствует санитарная обработка строк, что делает систему уязвимой для XSS-атак.
```
Добавил обработку минимальных/максимальных значений. Настроил фильтр для строковых полей, который делает `trim()` и `strip_tags()`. Здесь также можно использовать `HtmlPurifier`, чтоб оставлять безопасные теги у некоторых полей (предположим, на фронте может быть Rich Text редактор для некоторых полей).

```
Целостность данных. Отсутствие контроля транзакционности в операциях, затрагивающих несколько таблиц. Это создает риск нарушения консистентности данных при возникновении ошибок во время выполнения связанных запросов INSERT.
```
Добавил `UnitOfWorkInterface` в `CarService`. Теоретически это может и нарушает чистый код, однако с практической стороны сама имплементация транзакций скрыта от `CarService`. Данный интерфейс реализуется в `app\common\DbUnitOfWork`.

## Подготовка
Выполнить следующие команды:
- `git clone https://github.com/Yoowhi/yii2-itlink.git`
- `cd yii2-itlink`
- `cp .env.example .env`

После чего заполнить свежесозданный `.env` файл своими переменными. Предварительно заполненные переменные можно оставить как есть.

## Запуск
Выполнить `./run.sh`. Для остановки контейнера нажмите Ctrl+C.

## Жизненный цикл запроса:
1) Запрос поступает в контроллер
2) Контроллер прогоняет тело через модель для валидации
3) Модель на выходе возвращает DTO
4) DTO отправляется в сервис
5) Сервис в процессе работы передает в репозиторий/забирает из репозитория сущность
6) Репозиторий вызывает дата маппер
7) Дата маппер совершает действие с источником данных (в данном случае Postgres)
8) Все в обратном порядке возвращается в контроллер
9) Контроллер формирует тело ответа и возвращает его

## Решения, которые стоит прокомментировать
- В реальном проекте, возможно, стоило бы обернуть cars в отдельно взятый модуль (хотя бы ради удобного хранения файлов, связанных с этим модулем). Однако посмотрев URL в задании, я позволил себе использовать корневой модуль - Application.
- Регистрацию/аутентификацию/авторизацию не делал, поскольку в тестовом задании не было данного требования.
- Решил реализовать дата мапперы вместо использования готовых. Подробности ниже.
- Возможно, стоило бы наследоваться от BaseObject для более удобной инициализации свойств сущностей или DTOшек. В данном примере я решил их не использовать.
- Я намеренно не использовал ActiveRecord.
- Car и CarOption действуют как агрегат.
- Уже после коммита миграций я увидел некоторые недочеты. В случае этого тестового задания я позволил себе отредактировать миграцию вместо создания новой.
- Я сильно задержался пока пытался красиво реализовать дата мапперы (можно увидеть по времени коммитов, приходилось много раз откатываться на несколько шагов), по итогу получился абстрактный класс и пара трейтов на запись/чтение. Возможно оверинжиниринг, но мне было весело.
- **Я не использовал нейронки.**

## Data Mapper
Есть базовый абстрактный класс `DataMapper`, а также два трейта для записи и чтения: `WriteMapperTrait` и `ReadMapperTrait` соответственно. Дочернему дата мапперу необходимо определить эти два метода:
```php
protected abstract function getConnection(): Connection; // 
protected abstract function tableName(): string; 
```

##### При использовании `ReadMapperTrait` необходимо определить следующие методы:
```php
/**
 * Возвращает массив в виде таблиц и соответствующих им столбцам.
 * При наличии join надо указать все участвующие таблицы
 * @return ['tableName' => ['column1', 'column2']]
 */
abstract protected function selectColumns(): array;
/**
 * Метод для гидрции сущности.
 * Для автоматической гидрации можно вызвать mapToEntity(string $prefix, array $array, $entityClass)
 * где $prefix это имя таблицы из которой берутся данные. $entityClass это класс сущности.
 * mapToEntity() можно вызывать несколько раз для заполнения из разных таблиц или в разные сущности.
 */
abstract protected function arrayToEntity(array $array);

```
И при наличии join запросов переопределить следующий метод:
```php
/**
 * При join запросах переопределить данный метод и добавить в $query необходимый join
 */
protected function withRelations(Query $query);
```

##### При использовании `WriteMapperTrait` необходимо определить следующий метод:
```php
/**
 * Должен вернуть стандартный для Command массив с данными
 */
abstract protected function entityToArray($entity): array;
```
